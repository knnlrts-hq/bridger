Br i d ger Insight XG
LexisNexis® Bridger Insight® XG (XG) is a risk management product that lets you use search services
to compare your input entities against public and proprietary data. This functionality helps you lower
compliance costs and protect yourself from regulatory risk.
As part of the search, you tell XG which settings to use, who to assign the alerts to, and whether to store
the results in the XG database.
You also use XG to complete case management and workflow actions on the alerts and to manage batch
runs. A batch run contains the details about a batch search or an alert import and any associated alerts.
You can query, view, update, and delete alerts that are stored in the XG database. For example, you can
change the alert assignment or add an alert to an accept list.
In addition, you use XG to manage accept lists and custom screening lists. Accept lists can help you
suppress list screening matches in new searches that you have already processed and determined to be
false positives or determined that updates to the entity records need to be monitored. Custom screening
lists can help you flag entities in list screening searches that are unique to your organization.

Searchable Data
XG lets you search either structured input data or unstructured input data.
Structured data is organized into defined fields that lets XG treat like information the same for more
relevant results. You can use custom formats to search business entity records, individual entity records,
or unknown entity records. You can also use the following corresponding default formats to search
payment transaction data:

For information about ACH record content and format, see https://www.nacha.org. For information

CustomerSupportHub/s/article/2093.

information about Fedwire file processing in XG, see the related article that is available on the

Request Authentication and Credentials
The REST API uses a time-limited JWT (JSON Web Token) to authenticate requests.
Depending on your web service configuration, you request the JWT from XG or from your organization's
Azure AD client. You pass the token to XG web services in the Authorization header in each request
under the Bearer authorization scheme. Depending on the settings for the JWT issuer and the frequency
of the web service requests, a new JWT may be required for each request.
JWT Issued by XG
By default, the REST API uses a JWT that is issued by XG. You pass the XG user ID, client ID, and
password for the user who makes the request in the Authorization header of the Issue method. The
Issue method returns a JWT in the access_token property of the OAuth2Token class.
JWT Issued by Azure AD
XG Enterprise clients that use Azure AD for identity federation can configure the REST API to accept a
JWT that is generated by Azure AD instead of XG. Clients must set up their own Azure AD client and a
method to request the JWT from Azure AD.
®

LexisNexis Risk Solutions does not assist with Azure AD setup.
If you use Azure AD to issue the JWT, then you need to pass the base64-encoded XG client
ID and user ID separated by a forward slash in the X-ClientInfo header in each REST API
request. For example, an XG user with the client ID XGClient and the user ID Admin1 uses
the client info parameter XGClient/Admin1. The authorization parameter base64-encoded is
WEdDbGllbnQvQWRtaW4x.
For some methods, you can also pass the client ID and user ID in the ClientID and UserID properties
of the ClientContext class. If you also pass the credentials in the X-ClientInfo header, then the REST
API ignores these properties in the ClientContext class.

Chapter 6

L is t M ai n tenance Interface
LexisNexis® Bridger Insight® XG (XG) web services provide a ListMaintenance interface that you can use
to manage accept lists and custom screening lists.
Accept lists use information that you added from previous XG searches to filter out list screening
matches in new searches that you have already processed. Your organization has likely already
determined that those matches were false positive or has determined that updates to the entity records
in the match need to be monitored.
For more information about how accept lists work, see the Accept Lists and White List technical
white paper that is available on the LexisNexis® Risk Solutions Customer Support Hub at https://
lnrs.my.site.com/CustomerSupportHub/s/article/1969.
Custom screening lists are used to flag countries or entities in list searches for purposes that are unique
to your organization.

List Management
The ListMaintenance interface lets you manage accept lists or custom screening lists.
The ListMaintenance interface lets you perform the following actions on the lists:
• Add a list.
• Delete a list.
• Index a custom screening list.
Lists that are set to automatically index do not index when changes are made through XG
web services. Before changes are reflected in XG, the list must be manually indexed in the XG
application or using the IndexList method.
• Search for a list that contains specific information.
• Update a list.
• View a list.

AddList Method
The AddList method creates a new accept list or custom screening list.
The method is sent with the following HTTP request properties:
Request Line
The request uses the POST HTTP method. The request is sent to the following endpoint on the host
server: /api/ListMaintenance/AddList.
Required Headers
Authorization
The request uses a bearer JWT (JSON Web Token) for authentication.
Content-Type
The request contains a JSON (JavaScript object notation) message.
Host
The request is sent to https://<Your XG Environment>/LN.WebServices.
X-API-Key
The request contains the API (application programming interface) authentication key for your
client instance. XG Enterprise clients generate their API key in Enterprise Manager. XG Service
clients receive an API key from LexisNexis® Risk Solutions.
X-ClientInfo
This header is not required for Bridger JWT authentication. If the request uses a JWT that was
generated by Azure AD, then the request can contain the base64-encoded XG client ID and user ID
separated by a forward slash in the X-ClientInfo header. If the request uses this header, then XG
ignores the ClientID and UserID elements in the ClientContext class.
Body
The request contains the AddListRequest class.
Return Values
The response returns the system primary key that identifies the list.
If the method fails or generates an error, then the response returns an Error class.
